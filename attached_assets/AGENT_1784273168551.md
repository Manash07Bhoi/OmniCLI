# AGENT.md — Build Instructions for Autonomous Coding Agents

**Project:** OmniCLI · **Spec:** `PRD.md` (read it first — this file tells you *how* to build what that file specifies)
**Primary implementer:** Google Jules (autonomous) · **Reviewer:** Claude (architect) · **Product owner:** Roshan

---

## 1. The One Rule That Overrides Everything Else

**No mock, placeholder, dummy, or fake data or logic — anywhere, in any code path that ships.** This is not a style preference; treat it as a build-blocking requirement with the same severity as a failing test.

Concretely, this means:
- Never write a function that returns a hard-coded value "to make the caller compile" while the real logic is unfinished. Instead: return `Err(OmniError::NotYetImplemented { module, verb })` and leave a `// TODO(phase-N): <exact missing piece>` comment.
- Never fabricate sample output, fake success responses, or synthetic "looks realistic" data in a shipped path. Test fixtures are real (small) files, real SQLite databases, real archives — not synthetic stand-ins pretending to be real data.
- Never implement a feature against an in-memory fake and call it done. If a Port (§8.2 of the PRD) needs a Filesystem/Database/Network adapter, implement the *real* adapter. In-memory test doubles exist only behind `#[cfg(test)]` and are never reachable from `main()`.
- If you are not sure whether something counts as a placeholder, it does. Stop and implement the real thing, or mark it explicitly unimplemented — never ship something in between that merely *looks* finished.

Before marking any task complete, re-read the diff and ask: *if the user ran this exact command against their real files right now, would the output be genuinely computed from their data?* If the answer is anything but a clean yes, it's not done.

## 2. Build Order

Follow `PRD.md` §2.2 exactly. Do not start a Phase-2 module before every Phase-1 module has met the Definition of Done (§4 below) on all three target platforms. Do not add a language from §9 of the PRD before the phase that introduces it — e.g., no Lua code before Phase 2 begins, no WASM/Python before Phase 3.

1. **Phase 1 — Core:** `omni-core`, `omni-file`, `omni-search`, `omni-archive`, `omni-convert`, `scripts/install.sh`
2. **Phase 2 — Power Layer:** `omni-dev`, `omni-backup`, `omni-config`, `omni-workspace`, Lua scripting hooks
3. **Phase 3 — Ecosystem:** `omni-plugin` (WASM + Python runtimes), `omni-project` (`new`), package manager (`install`/`update`/`uninstall`), `omni-shell`
4. **Phase 4 — Reach:** docs (mdBook), optional Android companion

---

## 3. Repository Layout

Use exactly the structure in `PRD.md` §8.3. One crate per module under `crates/`. Do not create a module's crate until its phase begins — an empty crate stub committed early invites exactly the kind of "looks started but isn't real" state Rule 1 prohibits.

## 4. Definition of Done (per module)

A module is not complete until **all** of the following are true:

- [ ] Every verb listed for it in `PRD.md` §7 is implemented against real data sources
- [ ] `cargo clippy -- -D warnings` passes clean on the crate
- [ ] No `unwrap()`/`expect()` outside `main.rs` or `#[cfg(test)]`
- [ ] At least one happy-path and one failure-path integration test per verb, run against real fixture data in `tests/fixtures/`
- [ ] Tests pass in CI on Termux, Kali, and ParrotOS base images
- [ ] `--json` output mode verified to contain zero ANSI codes
- [ ] `--help` output matches the actual implemented flags exactly (regenerate, don't hand-maintain, if the CLI framework supports it)
- [ ] Errors are typed (`thiserror` enum for the module) and surfaced with an actionable message (`PRD.md` §15)

If any box is unchecked, the module is not done — do not report it as complete, do not move to the next phase.

## 5. Language-Specific Standards

**Rust (primary language, all phases)**
- Edition 2021+, workspace-level `Cargo.toml` with shared dependency versions.
- `clap` (derive API) for CLI parsing; `serde` for JSON/YAML/TOML; `rusqlite` for SQLite; `anyhow` at the CLI boundary, `thiserror` per-module.
- Format with `cargo fmt` before every commit.

**Lua (Phase 2, embedded via `mlua`)**
- Lua code is *user-facing scripting*, not core logic — core logic stays in Rust. Lua scripts call into a deliberately small, documented API surface exposed from Rust; do not let Lua reach into internals not explicitly exported.

**Python (Phase 3, plugin SDK)**
- Plugins communicate with the core over a documented JSON stdin/stdout protocol. Type-hint all public SDK functions. The SDK itself ships with real working example plugins, not stub templates with `pass` bodies.

**Bash (`scripts/install.sh`)**
- POSIX-compatible where feasible; must run correctly on Termux's shell environment, not just standard Debian bash. Test on an actual Termux session before merging changes to this script.

## 6. Security Rules

- Never hand-roll cryptography. Encryption uses `age` (PRD §12) — no custom cipher implementations, ever.
- Never add a network call that isn't explicitly required by the command being implemented (PRD §12 — zero telemetry, no phone-home). If a task seems to need one that isn't in the PRD, stop and flag it rather than adding it silently.
- Never commit secrets, API keys, or tokens — including in test fixtures.
- Plugin sandboxing (Phase 3) must use `wasmtime` with the capability manifest from `plugin.toml` (PRD §10.3) enforced, not merely documented.

---

## 7. How to Handle Ambiguity

If the PRD doesn't specify something precisely enough to implement:
1. Check whether a nearby section resolves it by implication (e.g., a flag pattern used elsewhere in §6).
2. If still ambiguous, make the smallest reasonable assumption, implement it, and state the assumption explicitly in the PR description under an "Assumptions" heading.
3. Never silently guess and hide the guess — an undocumented assumption is how mock-flavored logic sneaks in unnoticed.

---

## 8. Prohibited Patterns (explicit anti-patterns)

- `// mock implementation` or any comment implying shipped code is a stand-in
- Hard-coded "success" returns that don't reflect an actual operation performed
- `TODO` comments without a phase number and a specific description
- Test assertions like `assert!(true)` or tests that don't exercise real fixture data
- Copy-pasted `--help` text that doesn't match the actual flag parser
- A `new <template>` scaffold (PRD §7.10) that doesn't actually compile/build once generated

---

## 9. Git / PR Conventions

- Conventional commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`) — enforced in CI (PRD §18/§20).
- One PR per module-verb or tightly related group of verbs — not one PR per phase.
- PR description includes: what was implemented, what was tested (and against what real fixtures), and any "Assumptions" per §7 above.
