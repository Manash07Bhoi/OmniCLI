# OmniCLI — Product Requirements Document

**Version:** 1.0 · **Status:** Draft for build · **Owner:** Roshan (Product Owner / Technical Director) · **Architect:** Claude · **Implementation:** Google Jules (autonomous)
**Companion document:** `AGENT.md` — build instructions, coding standards, and the enforceable no-mock-data rules for any agent (Jules or otherwise) implementing this spec. Read `AGENT.md` before writing a single line of code.

> OmniCLI is a CLI operating platform for Termux, Kali Linux, and ParrotOS: one binary (`omni`), one command grammar, twelve modules, replacing a dozen single-purpose tools with a single coherent surface. This document specifies the full end-state vision **and** a disciplined, phase-gated path to it — every idea in the original brief is preserved; only the *order* and the *language count* have been made buildable by one developer working from an Android device.

## 1. Executive Summary & Vision

**Vision statement:** *A pentester, sysadmin, or developer working entirely from a terminal — on Termux, Kali, or ParrotOS — should never need to remember which of twelve different tools does what they want. They type `omni`, a module name, and a verb. Every module looks, behaves, and outputs the same way.*

OmniCLI unifies ten previously-separate tool concepts (file operations, universal search, a developer toolkit, a smart interactive shell layer, format conversion, backup, project scaffolding, config management, archiving, and a terminal workspace) into one binary with one grammar:

```
omni <module> <verb> [args] [flags]
```

The closest real-world precedent for this shape of project is **BusyBox**: it did not launch as a full coreutils replacement. It began as a single small rescue-disk utility and grew, applet by applet, over many years, into the multi-call binary it is today. OmniCLI follows the same growth model deliberately (see §2.2) rather than attempting Git-, Docker-, or Cargo-scale scope in a single build.

**One-line pitch:** *One CLI for file operations, search, conversion, backup, archiving, and workspace management — built for Termux, Kali Linux, and ParrotOS.*

## 2. Product Strategy

### 2.1 Strategic Reality Check (ADR)

This is a decision record, not a critique. Each row is a place where the original brief is preserved in intent but changed in execution, with the reasoning made explicit so it can be revisited later.

| # | Original ask | Decision | Reasoning |
|---|---|---|---|
| 1 | 500k+ GitHub stars as a target | Removed as a spec requirement; kept as an aspirational, non-binding outcome | As of this writing the #1 all-time GitHub repo (`build-your-own-x`) sits at ~525k stars — and it's a curated learning list, not software. The global top-100 (any category, all-time) threshold is ~120k. Best-in-class single-purpose terminal tools (e.g. `lazygit`) run ~67k. Designing *for* a star count produces marketing decisions, not product decisions. Designing for genuine daily-driver utility is what produces stars as a side effect. |
| 2 | 18-language/format stack | Trimmed to ~6 justified languages/formats, full mapping in §9 | The brief's own principle — *"not because more languages = better, each language must solve a specific problem"* — isn't satisfied by having Rust, Go, Zig, C, and C++ all nominally own "core/performance/systems." That's five toolchains competing for one job. Every category-leading tool in this exact space (`ripgrep`, `fzf`, `bat`, `starship`, `lazygit`) ships a **single** core language plus a shell installer. |
| 3 | PowerShell / Windows installer | Deferred indefinitely | Stated target platforms — Termux, ParrotOS, Kali — are all Linux-family. If Windows matters later, WSL2 already runs the Linux binary unmodified; that's a lower-cost path than maintaining a second native build + installer. |
| 4 | All 10 ideas as one v1 build | All 10 preserved as modules; delivered across 4 phases (§2.2) | Solo + Android + autonomous-agent implementation is a real constraint, not a limitation to argue away. A phase-gated build also directly serves the "no mock data" requirement (§19): a module isn't started until the one before it is real, tested, and shipped — there's no incentive for an agent to stub something to hit an arbitrary all-at-once deadline. |
| 5 | 150–250 page PRD | This document optimizes for density over page count | Padding a spec to hit a page target produces the same failure mode as mock data in code: content that *looks* complete but isn't load-bearing. Every section below is copy-paste usable. |

**What did not change:** all 10 original tool concepts, the `omni` command root, the "beautiful, colorful, styled output" requirement, the dual architecture (layered + hexagonal), the zero-mock-data rule, and the ambition level. Nothing was cut — it was sequenced.

### 2.2 Phased Roadmap

| Phase | Codename | Modules shipped | Language surface added | Exit criteria |
|---|---|---|---|---|
| **1** | Core | `file`, `search`, `archive`, `convert` | Rust + Bash (installer) | All 4 modules pass their test suites on Termux, Kali, and ParrotOS; single static binary < 12 MB |
| **2** | Power Layer | `dev`, `backup`, `config`, `workspace` | + Lua (embedded scripting) | Incremental backup verified against real filesystem trees; `omni shell` history/session persistence works across reboots |
| **3** | Ecosystem | `plugin`, `install`/package manager, `new`/project generator | + WASM (plugin sandbox), Python (plugin SDK) | A third-party plugin can be installed, sandboxed, and executed with no core-binary changes |
| **4** | Reach | Docs site, optional Android companion | + Markdown/mdBook, Java (optional) | Public docs live; README + repo metadata (§25) published |

This mirrors exactly how StreamPro grew (v2 → v4, scope expanding at each version) and how BusyBox itself grew — the difference here is the phase boundary is decided *before* the build starts, not discovered by scope creep mid-build.

## 3. Goals & Non-Goals

**Goals**
- Replace 10+ single-purpose CLI utilities with one coherent, memorable command grammar.
- Ship a single static binary with zero runtime dependency friction on Termux (no `apt`/root required for the core).
- Every command supports both a human-readable, styled output mode and a `--json` machine-readable mode.
- Every module is, internally, a plugin — including first-party ones — so the plugin API is proven by the core itself (dogfooding).
- Real, working implementations only, verified against real filesystems/databases — never mocked (§19).

**Non-goals (explicitly out of scope)**
- Replacing a full shell (bash/zsh/fish) — `omni shell` is a session/workspace layer *on top of* the user's existing shell, not a shell replacement.
- GUI or TUI-heavy file manager (that's a different product; OmniCLI stays command/flag driven, with optional interactive prompts only where destructive actions need confirmation).
- Native Windows binary or PowerShell installer (§2.1, row 3) — WSL2 compatibility is sufficient.
- Cloud-hosted service of any kind in Phases 1–3 (no server, no account, no telemetry) — everything is local-first.

## 5. Use Cases

1. Find every file over 50 MB modified in the last 7 days across a project, then compress the matches: `omni file find --size +50M --modified 7d | omni archive create backup.tar.gz`
2. Search across mixed content — code, PDFs, logs, a SQLite DB — for a CVE reference in one query: `omni search "CVE-2026-1234" --in code,logs,pdf`
3. Take an incremental, verified backup of a project directory before a risky operation: `omni backup create --name pre-deploy --source . --dest ~/omni-backups --incremental`
4. Decode and verify a JWT during an engagement without leaving the terminal: `omni dev jwt <token> --decode`
5. Scaffold a new Rust CLI project inside a Termux session: `omni new cli myproject --lang rust`
6. Convert a captured config from YAML to TOML for a target system: `omni convert config.yaml config.toml`

## 6. Unified Command Surface (CLI UX)

**Grammar:** `omni <module> <verb> [target] [--flags]`

Global flags (apply to every module):

| Flag | Effect |
|---|---|
| `--json` | Structured, machine-readable output; disables all styling |
| `--no-color` | Disables styled output (also respects the `NO_COLOR` env var automatically) |
| `--quiet` / `-q` | Suppresses non-error output |
| `--verbose` / `-v` | Adds debug-level tracing |
| `--dry-run` | For any destructive verb (clean, sync, restore, uninstall) — shows the plan without executing |
| `--config <path>` | Override the default `~/.config/omni/omni.toml` |

Full module → verb table:

| Module | Verbs |
|---|---|
| `file` | `find`, `copy`, `move`, `compare`, `duplicate`, `clean`, `compress`, `hash`, `encrypt`, `sync` |
| `search` | `<query>` (default action), `index`, `index --rebuild` |
| `dev` | `hash`, `json`, `yaml`, `base64`, `jwt`, `regex`, `uuid`, `qr`, `diff`, `sqlite` |
| `convert` | `<input> <output>` (format inferred from extension), `--list` |
| `archive` | `create`, `extract`, `list`, `convert` |
| `backup` | `create`, `restore`, `sync`, `verify`, `list` |
| `config` | `get`, `set`, `validate`, `convert` |
| `workspace` | `note`, `todo`, `snippet`, `session` |
| `shell` | *(no args — enters interactive mode)* |
| `new` | `rust`, `go`, `react`, `java`, `cli` |
| `plugin` | `list`, `install`, `remove`, `dev init` |
| `install` | `<module\|plugin-name>`, `update`, `uninstall` |

> Naming note: the original brief used both `Omni X` and `core X` in different sections. This PRD standardizes on `omni` as the single binary/root command everywhere, since that's the project name and the more explicit instruction. `core` is used only internally (crate names, §8.3), never in user-facing commands.

## 7. Module Specifications

Each module below is a first-party plugin conforming to the same Plugin API defined in §11 — there is no special-cased "built-in" code path.

### 7.1 `omni file` — File Operations (Phase 1)
- `find <pattern>`: supports `--type f|d|l`, `--size +N|-N` (K/M/G suffixes), `--modified <duration|range>`, `--path <dir>`
- `copy`/`move`: `--recursive`, `--verify` (post-copy hash comparison)
- `compare <a> <b>`: byte-diff or `--hash-only` for large files
- `duplicate --scan <dir>`: content-hash based duplicate detection, `--delete-dupes` with confirmation
- `clean`: `--older-than <duration>`, `--empty-dirs`, always dry-runnable
- `compress <path>`: delegates to `archive` module internally (no duplicated compression logic)
- `hash <path>`: `--algo sha256|blake3|md5`
- `encrypt <path>`: age-based asymmetric encryption, `--recipient <age-public-key>`
- `sync <src> <dst>`: `--watch` (live sync via filesystem events), `--delete-extraneous`

### 7.2 `omni search` — Universal Search (Phase 1)
- Default verb is a bare query: `omni search "<text>"`
- `--in files,pdf,code,sqlite,json,logs,zip` restricts content types searched
- `--regex`, `--case-sensitive` for literal control
- Backed by a persistent SQLite FTS5 index (§10.1); `search index --rebuild` forces a full re-crawl
- Structured queries pass through unmodified — e.g. `omni search "CVE-2026-1234" --in code` is literally the query text, not a special CVE mode; this keeps the search engine general rather than hard-coding domain-specific parsers

### 7.3 `omni dev` — Developer Toolkit (Phase 2)
- `hash`, `json` (`--pretty`, `--minify`, `--query <jq-style expr>`), `yaml` (`--to toml|json`), `base64` (`--decode`), `jwt` (`--decode`, `--verify --secret <key>`), `regex <pattern> <text>`, `uuid` (`--v4`, `--v7`, `--count N`), `qr <text> --out file.png`, `diff <a> <b>`, `sqlite <db> "<SQL>"`
- Every subcommand is stdin/stdout composable (Unix philosophy) so these chain with other modules and with the user's existing shell

### 7.4 `omni convert` — Universal Converter (Phase 1)
- Format inferred from file extensions: `omni convert report.pdf report.txt`
- Supported pairs at Phase 1: pdf↔txt, csv↔json, yaml↔toml, png↔webp, md↔html
- `convert --list` prints the live supported-pair matrix (never hard-coded in docs, always queried from the actual codec registry — this is itself a no-mock-data requirement: the help text and the implementation cannot drift apart)

### 7.5 `omni archive` — Universal Archive Tool (Phase 1)
- `create <out> <files...>`: format from extension (zip, tar, tar.gz, tar.xz, 7z)
- `extract <archive>`: auto-detects format from magic bytes, not just extension
- `list <archive>`: contents without extracting
- `convert <in> <out>`: re-containerize (e.g. zip → tar.xz) via stream, not full extract-then-recompress where the format allows it

### 7.6 `omni backup` — Incremental Backup (Phase 2)
- `create --name <job> --source <dir> --dest <dir> [--incremental]`: content-hash based change detection, not just mtime (mtime is used as a fast pre-filter, hash confirms)
- `restore --name <job> [--snapshot <id>] [--to <dir>]`
- `sync --name <job>`: re-runs a job's most recent config
- `verify --name <job> [--snapshot <id>]`: re-hashes stored files against the manifest and reports drift
- `list`: shows jobs and their snapshot history from `backup_jobs`/`backup_snapshots` (§10.1)

### 7.7 `omni config` — Config Manager (Phase 2)
- Distinct from `convert`: `config` is schema-aware (validates against a declared schema, supports env-var overlays and `get`/`set` on a live app config), while `convert` is raw format transcoding with no schema knowledge
- Formats: JSON, YAML, TOML, XML, INI
- `validate <file> --schema <schema>`: fails with line-numbered errors, never a silent pass

### 7.8 `omni workspace` — Notes, Todos, Snippets, Sessions (Phase 2)
- Merges the original "Smart Terminal" and "Terminal Workspace" ideas — they described the same underlying state (sessions, history) from two angles. `workspace` owns persistent state; `shell` (§7.9) is the interactive layer that reads/writes it.
- `note add "<title>" [--body <text>]`, `todo add "<desc>" [--due <date>]`, `snippet save <name> --lang <lang>`, `session list|resume <id>`

### 7.9 `omni shell` — Smart Interactive Shell (Phase 3)
- Entering `omni shell` opens a session-aware REPL layered on top of the user's existing shell (not a replacement)
- Features: searchable history (backed by `shell_history`, §10.1), bookmarks (`bm add`/`bm go <name>`), session-scoped variables, and a plugin hook point for `omni plugin`-installed shell extensions

### 7.10 `omni new` — Project Generator (Phase 3)
- `new rust|go|react|java|cli <name>`: each template is a real, functioning starter (compiles/builds out of the box), not a folder of TODO comments — templates are themselves subject to the no-mock-data rule (§19)

### 7.11 `omni plugin` — Plugin Management (Phase 3)
- `list`, `install <name|url>`, `remove <name>`, `dev init` (scaffolds a new plugin against the public Plugin API)
- Full spec in §11

### 7.12 `omni install` — Package Manager (Phase 3)
- `install <module>` (e.g. `omni install pdf`, `omni install image`): fetches an optional feature module not compiled into the core binary
- `update [module]`, `uninstall <module>`
- This is how the core binary stays under the Phase-1 size target (§13) while still supporting a large total feature surface

## 8. System Architecture

Two views are specified because they answer different questions: §8.1 answers *"what happens when a command runs"* (runtime/temporal view); §8.2 answers *"how is the code organized so business logic never depends on a specific filesystem or database"* (structural view). They describe the same system.

### 8.1 Layered View (Clean Architecture-flavored)

```
┌───────────────────────────────────────┐
│  CLI            (clap: parse & validate) │
├───────────────────────────────────────┤
│  Command Dispatcher   (routes module+verb → handler) │
├───────────────────────────────────────┤
│  Plugin Manager  (resolves first-party + third-party) │
├───────────────────────────────────────┤
│  Service Layer   (FileService, SearchService, ...) │
├───────────────────────────────────────┤
│  Core Engine     (hashing, compression, crypto — shared primitives) │
├───────────────────────────────────────┤
│  Platform Layer  (OS abstraction: paths, permissions, signals) │
├───────────────────────────────────────┤
│  Operating System   (Termux / Kali / ParrotOS) │
└───────────────────────────────────────┘
```
**Dependency Rule:** each layer calls only the layer directly beneath it. The Core Engine never imports from the Service Layer; the Platform Layer never imports from the Core Engine. Violating this in either direction is a code-review blocker (§19).

### 8.2 Hexagonal View (Ports & Adapters)

```
                    ┌────────────┐
                    │    CLI     │  ← driving adapter
                    └─────┬──────┘
                    ┌─────▼──────┐
                    │Application │  use-case orchestration
                    └─────┬──────┘
                    ┌─────▼──────┐
                    │   Domain   │  pure logic + Ports (traits)
                    └─────┬──────┘
                    ┌─────▼──────┐
                    │Infrastructure│ implements the Ports
                    └──┬───┬───┬──┬──┘
                 ┌──────┘  │   │  └───────┐
                 ▼         ▼   ▼          ▼
           Filesystem  Network  Database  Compression / Terminal
                (driven adapters)
```

**Reconciliation:** Service Layer (§8.1) ≈ Application; Core Engine (§8.1) ≈ Domain; Platform Layer (§8.1) ≈ Infrastructure/Adapters. The Domain layer holds trait definitions like `trait FileStore { fn read(&self, path: &Path) -> Result<Bytes>; }` — the Filesystem adapter implements it for real paths, and a test adapter can implement it in-memory for tests (this is the *only* place anything resembling "fake" data is acceptable: isolated unit-test doubles behind a Port, never shipped in a runtime code path — see §19).

### 8.3 Repository / Module Layout

```
omnicli/
├── Cargo.toml                    # workspace root
├── crates/
│   ├── omni-cli/                 # binary entrypoint, arg parsing, dispatcher
│   ├── omni-core/                # shared: hashing, compression, crypto, config
│   ├── omni-file/
│   ├── omni-search/
│   ├── omni-dev/
│   ├── omni-convert/
│   ├── omni-archive/
│   ├── omni-backup/
│   ├── omni-workspace/
│   ├── omni-config/
│   ├── omni-shell/
```

## 9. Technology Stack (ADR)

Every row from the original 18-item wishlist, with an explicit decision:

| Original item | Decision | Where it lives |
|---|---|---|
| Rust | **Kept — primary language** | Core engine + all first-party modules |
| Go | **Merged into Rust** | Networking handled by `tokio`; no separate toolchain |
| Zig | **Merged into Rust** | Performance-critical paths use `unsafe` Rust blocks where profiling justifies it, not a second language |
| C | **Merged into Rust** | OS abstraction via `std` + `libc` crate |
| C++ | **Merged into Rust** | Compression/indexing via `zstd`, `flate2`, `tantivy`-style crates |
| Python | **Kept — Phase 3** | Plugin SDK for third-party authors who want a higher-level language than Rust |
| Lua | **Kept — Phase 2** | Embedded via `mlua` (linked into the binary, no separate interpreter install) for `omni shell`/`omni workspace` user scripting |
| Bash | **Kept — Phase 1** | `scripts/install.sh` — zero-dependency bootstrap for Termux/Kali/ParrotOS |
| PowerShell | **Deferred indefinitely** | See §2.1 row 3 |
| TypeScript / JavaScript | **Deferred to Phase 4, replaced by Markdown** | Docs live as plain `.md` in-repo until Phase 4; if a site is built later, `mdBook` (Rust-based) keeps the toolchain consistent instead of introducing Node |
| HTML / CSS | **Deferred to Phase 4** | Generated by mdBook, not hand-authored |
| Java | **Kept — optional, Phase 4** | Android companion app, exactly as originally scoped ("optional") |
| SQLite | **Kept — reclassified** | This is a database engine, not a language; accessed via `rusqlite` (§10.1) |
| TOML | **Kept** | Primary config format (§10.2) — chosen because Cargo itself uses it, so the target audience (developers) already reads it fluently |
| YAML | **Kept — scoped to workflow files only** | Used specifically for `omni workspace` multi-step automation recipes (Phase 3), not general config — avoids "three config formats for the same job" |
| JSON | **Kept** | The `--json` machine-output contract on every command; plugin manifests may also be JSON |
| WASM | **Kept — Phase 3** | Plugin sandboxing target via `wasmtime`; plugins can be *authored* in Rust, AssemblyScript, C, etc. and compiled *to* WASM — it's a security boundary, not a language |

**Net result:** ~6 justified languages/formats carrying real, non-overlapping responsibilities, versus 18 nominal ones with 5-way overlap on "core/performance." This is the brief's own stated principle, applied strictly.

## 11. Plugin SDK & Package Manager

- **Design principle:** every module in §7 is implemented *as* a plugin against the same public API third-party developers use. There is no privileged "core" code path that plugins can't also use.
- **Trust levels:** `first-party` (shipped in the core binary, Phase 1–2), `verified` (audited, signed, Phase 3), `community` (unverified, requires `allow_community = true` and runs WASM-sandboxed only).
- **Runtimes:** `native` (first-party only, compiled into the binary), `wasm` (sandboxed, via `wasmtime`, for untrusted third-party plugins), `python` (subprocess with a JSON stdin/stdout protocol, for the higher-level Plugin SDK).
- **Package manager (`omni install`):** separate from plugin *trust* — this is how optional first-party feature modules (e.g., `omni install pdf`, `omni install image`) are fetched without bloating the core binary. `omni update` and `omni uninstall` operate the same registry (§10.1 `plugin_registry`).

---

## 12. Security Model

- **Encryption:** `omni file encrypt` uses `age` (modern, audited, asymmetric) — never a hand-rolled cipher.
- **Plugin sandboxing:** community plugins run under `wasmtime` with capability-based permissions declared in `plugin.toml` (§10.3) — filesystem and network access are opt-in per-path, not ambient.
- **Secrets:** never logged, never written to `shell_history`, never included in `--json` output by default (fields matching common secret patterns are redacted unless `--show-secrets` is explicit).
- **Network default:** OmniCLI makes zero network calls unless the command explicitly requires one (`plugin install`, `install <module>`, `backup sync` to a remote). No telemetry, no phone-home, ever — this is a hard requirement, not a Phase-1-only default.
- **Supply chain:** every plugin has a required `checksum_sha256` in the registry; installs verify before execution.

## 14. CLI Visual Design System

This directly implements the "beautiful, stylish, colourful, highlighted" requirement as an enforceable spec rather than a vibe.

**Color semantics**

|---|---|---|
| Success | `#2ECC71` | ✓ / `[OK]` |
| Error | `#E74C3C` | ✗ / `[FAIL]` |
| Warning | `#F1C40F` | ⚠ / `[WARN]` |
| Info | `#3498DB` | ℹ / `[INFO]` |
| Accent / highlight | `#9B59B6` | — |
| Muted / secondary | `#7F8C8D` | — |

**Rules**
- Always respect `NO_COLOR` and `--no-color`; always auto-disable styling when stdout is not a TTY (piped output must be clean, undecorated text unless `--force-color` is passed).
- `--json` mode never contains ANSI codes, ever, under any flag combination.
- Icons are Unicode with a pure-ASCII fallback detected via terminal capability probe — no emoji. This keeps output reading as a serious tool rather than a toy, and avoids font-rendering inconsistencies across Termux/Kali/ParrotOS terminal emulators.
- Any operation likely to exceed 500ms shows a progress indicator; list-style output (`backup list`, `plugin list`) renders as an aligned table with a bolded header row.

## 15. Logging & Error Handling

- Exit codes follow the standard convention: `0` success, `1` generic failure, `2` usage error, `126`/`127` for permission/not-found where applicable.
- Every error message states: what failed, why (if known), and the next actionable step — never a bare stack trace to a non-`--verbose` user.
- `--verbose` adds structured tracing (via `tracing` crate) to stderr; stdout stays clean for piping regardless of verbosity.
- All errors are typed (`thiserror` per-module error enums surfaced through `anyhow` at the CLI boundary) — no stringly-typed error handling.

---

## 16. Cross-Platform Strategy

- **Development happens on Termux (aarch64/Android)** — this is the builder's only machine. Local Termux builds are for day-to-day iteration and testing, not for producing release binaries.
- **Release binaries are built by CI**, not cross-compiled locally: GitHub Actions' free tier for public repos provides the full target matrix (x86_64 for Kali/ParrotOS, aarch64 for Termux/RPi-based Kali) without needing local cross-toolchains or a credit card. This is the practical unlock for a phone-only dev environment — the CI matrix does the cross-compilation, not the phone.
- Windows: not a native target (§2.1); WSL2 runs the Linux build unmodified if ever needed.

---

## 17. Testing Strategy

- **Unit tests** per crate, run against real temp-directory filesystems (via `tempfile`) — never against a mocked filesystem trait in a shipped code path; test doubles behind Ports (§8.2) are permitted *only* inside `#[cfg(test)]`.
- **Integration tests** run each module's actual CLI binary against real fixture data checked into `tests/fixtures/` (real small files, real archives, real SQLite DBs — not synthetic placeholders).
- **Cross-platform CI matrix** runs the full suite on Termux (via a Docker Android/Termux emulation image or act on aarch64 runners), Kali, and ParrotOS base images.
- **Definition of "tested"** for a module to exit its phase: all verbs have at least one happy-path and one failure-path integration test, running against real data, green in CI on all three target platforms.

## 18. CI/CD Pipeline

- GitHub Actions, free tier (public repo, no credit card required).
- On every PR: `cargo fmt --check`, `cargo clippy -- -D warnings`, `cargo test --workspace`, cross-platform build matrix (§16).
- On tag push (`v*`): build release binaries for all target triples, generate checksums, publish a GitHub Release with the binaries attached, update `omni update`'s version manifest.

## 19. Coding Standards & Quality Bar (No-Mock-Data Rule)

This section is the enforceable version of the project's hard requirement: **no mock, placeholder, dummy, or fake data or logic, anywhere, in any shipped code path.**

- Every function that returns data returns *real* data from a real source (filesystem, SQLite, or an actual computed value) — never a hard-coded stand-in "to make it compile."
- A feature that can't be finished in one pass is marked `// TODO(phase-N): <what's missing>` and left **unimplemented** (returns a clear `Err(NotYetImplemented)`), never faked with a plausible-looking fake return value.
- No `unwrap()`/`expect()` in library code paths (`main.rs` and test code are the only exceptions) — all fallible paths return `Result` with a typed error (§15).
- `cargo clippy -- -D warnings` must pass clean; warnings are not deferred.
- The full enforcement checklist — what an agent must verify before marking any module "done" — lives in `AGENT.md`, since that's the document actually read at implementation time.

## 20. Release Process & Versioning

- Semantic versioning (`MAJOR.MINOR.PATCH`). Phase 1 ships as `v0.1.0`; each phase boundary (§2.2) is a minor version bump; `v1.0.0` is reserved for the completion of Phase 2 (Core + Power Layer both stable).
- Changelog generated from conventional commits (`feat:`, `fix:`, `docs:`, etc. — enforced via commit-lint in CI).
- `omni update` checks the published version manifest and self-updates the binary in place.

## 22. Risks & Mitigations

| Risk | Mitigation |
|---|---|
| Solo-maintainer bus factor | Modular crate layout (§8.3) means individual crates can be community-co-maintained post-open-sourcing without touching the rest of the codebase |
| Scope creep re-inflating to the original 10-in-1/18-language shape | Phase gates in §2.2 are exit criteria, not soft targets — a phase isn't "started" until the prior one is CI-green on all 3 platforms |
| Star-count treated as a spec requirement | Reframed in §2.1 as a lagging indicator; actual tracked KPIs should be installs, issue engagement, and contributor count, not stars |
| Cross-compiling from an Android-only dev machine | §16 — CI does the cross-compilation, not local Termux |
| Untrusted plugin code execution | WASM sandboxing + capability manifest (§12), and deliberately not built until Phase 3 rather than rushed into v1 |
| Autonomous agent produces "looks-done" stub code to hit a deadline | §19 + `AGENT.md`'s Definition of Done checklist — a module can't be marked complete without passing tests against real data |
